# Conversion fidelity matrix

This document is the per-tag reference for what `@kleiobase/gedcom-converter` does when it converts between GEDCOM versions. It is the source of truth for whether a given structure round-trips cleanly, is degraded, or is dropped — and for which diagnostic code is emitted when that happens.

The matrix is generated by inspection of `src/mappings/551-to-v7.ts`, `src/mappings/v7-to-551.ts`, `src/mappings/date/*-to-*.ts`, and the post-mapper sanitisation layer in `src/gedcom551/compatibility.ts`. When this document and the code disagree, the code wins — please open an issue.

## Legend

| Outcome | Meaning |
| --- | --- |
| `clean` | Full semantic equivalence. No diagnostic emitted. |
| `lossy: <reason>` | Partial mapping — the target version cannot fully express the source. A diagnostic is emitted so callers can detect the degradation. |
| `_TAG` | Preserved as a `_`-prefixed custom extension. SCHMA declares it on the v7 side. |
| `dropped: <reason>` | Cannot be expressed in the target version. The structure is removed from the output. |
| `N/A` | The tag does not exist in the source version (e.g. `SNOTE` in 5.5.1). |

## Directions

- **v7 → 5.5.1**: GEDCOM 7.0.18 down-conversion. Combines the inverse mapper in `src/mappings/v7-to-551.ts` with the post-mapper sanitisation in `src/gedcom551/compatibility.ts` (which applies the bulk of 5.5.1-compliance rewrites).
- **5.5.1 → v7**: GEDCOM 5.5.1 up-conversion through `src/mappings/551-to-v7.ts`.
- **5.5 → v7**: Transitive — the converter does `5.5 → 5.5.1` (version bump only, no structural change) then `5.5.1 → v7`. The outcomes are therefore identical to the 5.5.1 → v7 column unless the 5.5 form differs lexically. Structures introduced in 5.5.1 (e.g. `EMAIL`, `FAX`) are treated as no-ops on the 5.5 side.

## How to read the tables

Each row is one tag. The "Notes" column flags context-specific behaviour (e.g. a tag behaves differently under `OBJE.FILE` than under `PLAC`). Multiple rows for the same tag mean the tag has materially different handlers in different parents.

## Pointer tags (general behaviour)

All pointer tags (`ALIA`, `ANCI`, `ASSO`, `CHIL`, `DESI`, `FAMC`, `FAMS`, `HUSB`, `NOTE`, `OBJE`, `REPO`, `SOUR`, `SUBM`, `WIFE`) share:

- v7 → 5.5.1: `clean` for pointer-to-existing-record; `lossy` (rewritten to a NOTE explaining what was referenced) when `@VOID@`. Diagnostic: `VOID_POINTER_NOTED`. Pointers to *missing* records are dropped with `DROPPED_MISSING_POINTER`.
- 5.5.1 → v7: `clean`.
- `HUSB` / `WIFE` / `CHIL` `PHRASE` children are hoisted into a per-pointer note on the inverse direction (`POINTER_PHRASE_NOTED`).
